# Brainfuck Interpreter — Research Report

**Date:** 2026-06-29 (original analysis)  
**Scope:** Full repository analysis, WASM web app failure investigation, and remediation plan.

> **Origin:** This project extends the CLI Brainfuck interpreter from [Learn Rust Basics By Building a Brainfuck Interpreter](https://blog.sheerluck.dev/posts/learn-rust-basics-by-building-a-brainfuck-interpreter/) (Mrsheerluck [learning-rust](https://blog.sheerluck.dev/series/learning-rust/) series) into a WASM-backed **Brain Fog Visualizer** with step/run/reset, tape view, and execution highlighting.
>
> **Status (2026-06):** Phases 1–3 and deployment are complete. Sections below describing broken-on-load behaviour, missing `src/pkg`, and stale defaults are **historical** — kept for context. See [`README.md`](README.md) for current features and roadmap.

## 1. Executive Summary

This repository is a **Brainfuck visual debugger** (“Brain Fog Visualizer”) built as a dual-target Rust project:

| Target | Technology | Role |
|--------|------------|------|
| CLI | Rust binary (`src/main.rs`) | Runs a hardcoded Hello World program |
| Web | React 19 + Vite 8 + Rust/WASM (`core-wasm`) | Interactive step/run visualizer |

The web app is the main product. It loads a `wasm-bindgen` module, constructs a `BrainfuckEngine`, and steps through Brainfuck one instruction at a time while rendering the tape, program counter (PC), and stdout.

**The web app is currently broken in multiple ways.** Reproducing with `npm run dev` and opening `http://localhost:5173/` produces immediate runtime crashes. The Vite dev server logs:

```
RuntimeError: memory access out of bounds
  > BrainfuckEngine.get_dp (core_wasm.js)
  > updateUI (App.jsx:51)
  > tick useEffect (App.jsx:46)
```

A follow-up error ~30 seconds later implicates the wasm-bindgen `FinalizationRegistry`:

```
RuntimeError: memory access out of bounds
  > BrainfuckEngineFinalization (core_wasm.js:95)
```

These are not build failures — `npm run build` succeeds. The problems are **runtime lifecycle bugs in React**, **invalid default Brainfuck code**, **missing bounds checks in the WASM API**, and **workflow gaps** (WASM artifacts not present on a fresh checkout).

## 3. How the System Works

### 3.1 Brainfuck Execution Model

Both interpreters implement a classic Brainfuck VM:

- **Tape:** 30,000 cells (`u8`, wrapping arithmetic on `+`/`-`)
- **Data pointer (`dp`):** index into tape
- **Program counter (`pc`):** index into program
- **Bracket map:** precomputed O(n) stack pass for O(1) `[` / `]` jumps

**Commands:** `><+-.,[]` (input `,` stubbed or missing depending on target)

### 3.2 CLI Interpreter (`src/main.rs`)

- Hardcoded Hello World program string
- Runs to completion in a single `while` loop
- **Bracket errors:** panics (`expect("Unmatched ]")`, `panic!("Unmatched [")`)
- **Pointer movement:** unbounded `dp += 1` / `dp -= 1` (can index out of tape bounds → panic)
- **Comments:** non-command bytes silently ignored via `_ => {}`
- **Run:** `cargo run`

### 3.3 WASM Engine (`core-wasm/src/lib.rs`)

Exposed to JavaScript via `#[wasm_bindgen]` as `BrainfuckEngine`:

| Method | Behavior |
|--------|----------|
| `new(program_text)` | Filter to BF chars only, build bracket map, init tape |
| `step()` | Execute **one** instruction; return `true` if more steps remain |
| `get_dp()` / `get_pc()` | Inspect VM state |
| `get_output()` | Clone accumulated `.` output as JS string |
| `get_tape_cell(index)` | Read tape[index] — **no bounds check** |
| `free()` | Generated by wasm-bindgen; releases Rust allocation |

**Semantic differences from CLI:**

| Behavior | CLI | WASM |
|----------|-----|------|
| Program source | Hardcoded | User-editable |
| Non-BF characters | Ignored at runtime | Stripped at parse time |
| `dp` on `>` | Unbounded increment | Wraps: `(dp + 1) % tape.len()` |
| `dp` on `<` at 0 | Underflow panic | Wraps to `tape.len() - 1` |
| Bracket mismatch | Panics | Silently skipped (no panic) |
| Execution model | Full run | One step per `step()` call |
| Input `,` | No-op stub | Not implemented (`_ => {}`) |

### 3.5 Web App Architecture (`src/App.jsx`)

```
┌─────────────────┐     init()      ┌──────────────────┐
│   App.jsx       │ ──────────────► │  core_wasm.js    │
│  (React UI)     │                 │  (wasm-bindgen)  │
└────────┬────────┘                 └────────┬─────────┘
         │ step() / getters                   │
         ▼                                    ▼
┌─────────────────┐                 ┌──────────────────┐
│  engineRef      │ ◄────────────── │ BrainfuckEngine  │
│  (useRef)       │   WASM linear   │  tape, dp, pc,   │
└────────┬────────┘   memory        │  bracket_map     │
         │                            └──────────────────┘
         ▼
┌─────────────────┐
│ React state:    │
│ dp, pc, output, │
│ tapeWindow      │
└─────────────────┘
```

**UI sections:**
- Header with PC and Run / Step / Reset controls
- Memory tape: 21-cell window centered on `dp`, with ASCII display for printable values
- Code editor: textarea (edits trigger immediate reset)
- Standard output: green-on-black console

**State sync pattern (fragile):** WASM state lives outside React. The app uses a `tick` counter incremented after each step/reset to force a `useEffect` that calls `updateUI()`. Comments in the code label this as `FIX 1` and `FIX 2`.

**Lifecycle:**
1. `init()` loads `.wasm` asynchronously → `setIsReady(true)`
2. `isReady` effect calls `handleReset()` → `new BrainfuckEngine(code)`
3. Step/Run calls `engine.step()` then bumps `tick`
4. Reset calls `engine.free()`, creates new engine, bumps `tick`
5. Run loop: `setInterval(handleStep, 5ms)` while `isRunning`

**React StrictMode** is enabled in `main.jsx`, which double-invokes effects in development.

## 4. Specificities and Design Decisions

1. **Steppable VM:** WASM engine is intentionally single-step for visualization; CLI runs atomically.
2. **Filtered parse in WASM:** Newlines and comments in the editor are stripped so they don't affect PC indexing.
3. **Circular tape in WASM:** `>` and `<` wrap at boundaries (unlike CLI).
4. **Manual memory management:** `handleReset` explicitly calls `engine.free()` before creating a new engine.
5. **No input support in web:** `,` is not implemented in WASM `step()`.
6. **Edition split:** Root crate uses Rust 2024; `core-wasm` uses 2021.
7. **wasm-opt disabled:** `wasm-opt = false` in `core-wasm/Cargo.toml` for faster builds.
8. **Duplicated logic:** Bracket-map and interpreter code are copy-pasted between `main.rs` and `lib.rs` with divergent behavior.

## 5. WASM Web App — Failure Analysis

### 5.1 Critical: Use-After-Free on Engine (Load-Time Crash)

**Symptom:** `RuntimeError: memory access out of bounds` in `BrainfuckEngine.get_dp()` called from `updateUI()` via the `tick` useEffect. Occurs immediately on first page load in dev mode.

**Root cause:** The React effect graph creates a race between engine destruction and UI reads.

```javascript
// isReady effect — runs handleReset (which calls free())
useEffect(() => {
  if (isReady) handleReset();
}, [isReady]);

// tick effect — reads engine via updateUI()
useEffect(() => {
  updateUI();
}, [tick]);

const handleReset = () => {
  if (engineRef.current) {
    engineRef.current.free(); // sets __wbg_ptr = 0 on JS wrapper
  }
  engineRef.current = new BrainfuckEngine(code);
  setTick((t) => t + 1);
};
```

**Failure mechanism:**

1. React StrictMode double-invokes effects in development.
2. First `handleReset` creates engine **A** and schedules `setTick`.
3. Second `handleReset` (strict re-run) calls `A.free()` — the JS wrapper still exists briefly with `__wbg_ptr = 0`.
4. The `tick` effect’s `updateUI()` can call `get_dp()` on a freed wrapper → WASM trap (`memory access out of bounds` in browser; `null pointer passed to rust` in Node).
5. Manual `free()` combined with wasm-bindgen’s `FinalizationRegistry` can produce a second crash in `BrainfuckEngineFinalization` when the GC later attempts cleanup on corrupted WASM state.

**Evidence:**
- Vite dev server terminal captured both error signatures on first load.
- Node simulation: `engine.free(); engine.get_dp()` → `null pointer passed to rust`.
- StrictMode is explicitly enabled in `src/main.jsx`.

### 5.2 Critical: Invalid Default Program (Infinite Loop)

**Default code in `App.jsx`:**

```brainfuck
>++++++++[<++++++++>-]<[.]
>+++++++[<++++>-]<+.
+++++++.
.
+++.
```

The first line ends with `<[.]`. After setup, cell 0 holds value 64 (`@`). The loop `[.]` prints the cell but **never decrements it**, so `]` always jumps back to `[` — **infinite loop**.

**Consequences:**
- Clicking **Run** spams `@` forever; output string grows without bound.
- Rust `String` in WASM eventually hits allocation limits → `capacity_overflow` panic (`unreachable` trap).
- UI becomes unresponsive due to 5ms step interval + full re-render on every tick.

**Verified:** Running the default program in Node exceeded 100,000 steps with monotonically growing `@` output. Changing `[.]` to `[.-]` allows the program to terminate and produce expected output.

### 5.3 High: `get_tape_cell` Out-of-Bounds Panic

`updateUI()` renders a 21-cell window:

```javascript
const startIdx = Math.max(0, currentDp - 10);
for (let i = startIdx; i < startIdx + 21; i++) {
  engineRef.current.get_tape_cell(i);
}
```

Rust implementation:

```rust
pub fn get_tape_cell(&self, index: usize) -> u8 {
    self.tape[index]  // no bounds check — panics if index >= 30000
}
```

When `dp >= 29990`, the loop requests indices 30000–30010 → Rust panic → WASM `unreachable` trap.

**Verified:** After 29,999 `>` instructions (`dp = 29999`), `get_tape_cell(30000)` panics.

### 5.4 High: Missing WASM Artifacts on Fresh Checkout

- `src/pkg/` is generated by `wasm-pack` and gitignored (`src/pkg/.gitignore` contains `*`).
- `npm run dev` does **not** run `wasm-build`.
- Fresh clone → `import "./pkg/core_wasm.js"` fails with module-not-found until `npm run wasm-build` is run manually (requires `wasm-pack` on PATH).

### 5.5 Medium: Fragile React ↔ WASM Sync

The `tick` counter workaround indicates React does not observe WASM mutations. This pattern:
- Adds unnecessary effect cycles
- Increases the window for use-after-free bugs
- Calls `get_output()` (full string clone) on every tick

### 5.6 Medium: Effect Dependency / Stale Closure Issues

| Issue | Location | Risk |
|-------|----------|------|
| `handleStep` not in run-loop effect deps | `[isRunning]` effect | Stale closure (mitigated by `engineRef`) |
| `handleReset` not in `isReady` effect deps | `[isReady]` effect | Stale `code` if effect re-fires without code dep |
| `updateUI` not in tick effect deps | `[tick]` effect | Lint warning; fragile |

### 5.7 Medium: Reset on Every Keystroke

```javascript
onChange={(e) => {
  setCode(e.target.value);
  handleReset();
}}
```

Each character typed frees and recreates the WASM engine, re-parses the program, and bumps `tick`. This is expensive and amplifies lifecycle bugs.

### 5.8 Low: Documentation and Tooling Gaps

- `README.md` is the default Vite template — no build/run instructions for this project.
- `BF.md` references `.vscode/launch.json` which is gitignored and absent.
- No `predev` / `postinstall` hook to auto-build WASM.
- No automated tests despite two divergent interpreter implementations.

### 5.9 Low: Semantic Inconsistencies (CLI vs WASM)

Maintaining two separate engines with different tape wrapping, parsing, and error handling will cause confusion when validating programs against the CLI.

## 7. Remediation Plan

### Phase 1 — Stop the Crashes (Priority: Immediate)

#### 1.1 Fix React ↔ WASM lifecycle

**Goal:** Eliminate use-after-free between `free()`, effect re-runs, and `updateUI()`.

**Changes in `App.jsx`:**

1. **Remove the `tick` workaround.** Call a `syncUi()` function directly at the end of `handleReset()` and `handleStep()` instead of routing through a `useEffect`.
2. **Stop calling `free()` manually** during normal reset, OR set `engineRef.current = null` immediately before `free()` and guard all reads:

   ```javascript
   const syncUi = () => {
     const engine = engineRef.current;
     if (!engine) return;
     // read state...
   };
   ```

3. **Add a mount guard** for WASM init to prevent StrictMode double-init races:

   ```javascript
   const initRef = useRef(null);
   useEffect(() => {
     if (initRef.current) return;
     initRef.current = init().then(() => setIsReady(true));
   }, []);
   ```

4. **Add cleanup on unmount** — null out `engineRef` and optionally free:

   ```javascript
   useEffect(() => {
     return () => {
       engineRef.current?.free();
       engineRef.current = null;
     };
   }, []);
   ```

5. **Consolidate reset logic** into a single `useEffect` keyed on `[isReady, code]` with debounce for editor changes, instead of resetting on every keystroke synchronously.

#### 1.2 Fix the default Brainfuck program

Change the infinite loop to a terminating print loop:

```diff
->++++++++[<++++++++>-]<[.]
+>++++++++[<++++++++>-]<[.-]
```

Verify the full default program produces sensible output when stepped and when run to completion.

#### 1.3 Add bounds safety to `get_tape_cell`

In `core-wasm/src/lib.rs`:

```rust
pub fn get_tape_cell(&self, index: usize) -> u8 {
    self.tape.get(index).copied().unwrap_or(0)
}
```

Alternatively, clamp the UI window in `App.jsx`:

```javascript
const tapeLen = 30000;
const endIdx = Math.min(startIdx + 21, tapeLen);
```

Prefer fixing in Rust so the API is safe for all callers.

### Phase 2 — Dev Workflow and Build Reliability

#### 2.1 Auto-build WASM before dev

```json
"scripts": {
  "predev": "npm run wasm-build",
  "dev": "vite",
  ...
}
```

Or use `concurrently` / `vite-plugin-wasm` if watch-mode Rust builds are desired.

#### 2.2 Document setup in README

Replace the Vite boilerplate README with:

```bash
# Prerequisites: Rust, wasm-pack, Node.js
npm install
npm run wasm-build   # or rely on predev
npm run dev
```

#### 2.3 Decide on `src/pkg/` versioning strategy

Either:
- **Commit built artifacts** (remove `src/pkg/.gitignore`, add pkg to repo), or
- **Keep generated** but enforce `predev`/`prebuild` hooks and document the requirement prominently.

### Phase 3 — Correctness and Robustness

#### 3.1 Implement `,` (input) in WASM

Add a JS callback import or an input buffer on `BrainfuckEngine` so interactive programs work.

#### 3.2 Add step limit / output cap

Prevent runaway execution:

```rust
const MAX_OUTPUT_LEN: usize = 1_000_000;
// in step(), refuse to grow output beyond cap
```

And/or expose `max_steps` in the UI with a stop condition in the run loop.

#### 3.3 Unify interpreter semantics

Extract shared logic into a `brainfuck-core` Rust library crate used by both `src/main.rs` and `core-wasm`. Pick one specification for:
- Tape wrapping behavior
- Comment handling
- Bracket error policy

#### 3.4 Add tests

| Test | Target |
|------|--------|
| `cargo test` in shared core | Bracket map, step semantics, I/O |
| WASM tests via `wasm-bindgen-test` | Engine API |
| Vitest component test | React lifecycle (no crash on mount) |
| Golden output test | Default program output matches expected string |

### Phase 4 — Polish

- Debounce editor resets (300ms) or add explicit “Apply” button.
- Remove dead code: `App.css`, unused `assets/`.
- Align Rust editions (both 2024 or both 2021).
- Add `.vscode/launch.json` or remove references from `BF.md`.
- Consider disabling StrictMode only as a last resort — fixing lifecycle is the correct solution.

## 11. Conclusion

The project is a well-scoped Brainfuck visualizer prototype with a clean stepping API, but the web app fails due to a **combination of React StrictMode lifecycle bugs around manual `engine.free()`**, an **infinite-loop default program**, and **missing bounds checks** in the WASM tape accessor. The build pipeline also omits WASM compilation from the dev workflow, making fresh checkouts non-functional.

The highest-impact fixes are small and localized: rework `App.jsx` state sync to eliminate the `tick`/`free()` race, correct the default Brainfuck snippet, and harden `get_tape_cell`. These three changes should restore basic functionality without architectural overhaul; the shared-core refactor and test suite are recommended follow-ups for long-term maintainability.
